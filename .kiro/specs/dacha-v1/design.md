# Design: –і–∞—З–∞ (dacha) v1

## Overview

–і–∞—З–∞ is a TypeScript CLI tool that runs on Deno. The user's dotfiles repo contains TypeScript configuration files that define the desired system state. The CLI evaluates these configs, resolves profiles and dependencies, and converges the system. A separate long-running daemon process handles dotfile sync and update checks.

## Technology Choices

- **Runtime**: Deno вАФ runs TypeScript directly, no build step, built-in file watching, good stdlib
- **Distribution**: `deno compile` to produce self-contained binaries per platform (no runtime dependencies)
- **CLI framework**: `@std/cli` (Deno stdlib) or Cliffy for subcommand parsing
- **Secret encryption**: age (via `age` CLI binary, already available in nixpkgs)
- **File watching**: Deno's `Deno.watchFs` for the sync daemon
- **Git operations**: shell out to `git` CLI (already required on the system)
- **Notifications**: shell out to `terminal-notifier` (macOS) / `notify-send` (Linux)
- **Daemon management**: launchd plist (macOS), systemd unit (Linux), generated by `dacha sync start`

## Detailed Design

### Core Types (types.ts)

```typescript
// Platform detection result
interface Platform {
  os: "darwin" | "linux";
  arch: "arm64" | "x64";
  distro?: string;           // "ubuntu", "fedora", etc. (linux only)
  packageManager: PackageManagerType;
}

type PackageManagerType = "brew" | "apt" | "yum" | "dnf";

// Base resource вАФ all resources extend this
interface Resource {
  id: string;                 // Unique identifier
  type: "package" | "dotfile" | "command" | "secret";
  dependsOn?: string[];       // Explicit dependency IDs
  outputs?: Record<string, string>;  // Named output values (populated after execution)
  contributedBy?: string;     // Profile that declared this resource (set during merge)
}

// Package resource
interface PackageResource extends Resource {
  type: "package";
  name: string;               // Canonical name
  brew?: string;              // Override for brew
  brewCask?: string;          // Install as cask instead
  apt?: string;               // Override for apt
  yum?: string;               // Override for yum/dnf
  onlyOn?: PlatformFilter;    // Only install on matching platforms
}

// Dotfile resource
interface DotfileResource extends Resource {
  type: "dotfile";
  source: string;             // Path relative to repo root
  destination: string;        // Absolute path or ~/relative
  template?: boolean;         // If true, interpolate {{output.id.key}} references
}

// Command resource
interface CommandResource extends Resource {
  type: "command";
  run: string;                // Shell command to execute
  check?: string;             // If exits 0, skip the run (idempotency)
  critical?: boolean;         // If true, halt apply on failure
  onlyOn?: PlatformFilter;
  captureOutput?: string;     // If set, stdout is captured as output with this key
}

// Secret resource
interface SecretResource extends Resource {
  type: "secret";
  source: string;             // Path to .age file relative to repo
  destination: string;        // Where to place decrypted file
  permissions?: string;       // e.g. "0600", defaults to "0600"
}

// Platform filter for conditional resources
interface PlatformFilter {
  os?: Platform["os"];
  arch?: Platform["arch"];
  distro?: string;
}

// Profile вАФ the unit of composition
interface Profile {
  name: string;
  extends?: Profile[];
  packages?: PackageResource[];
  dotfiles?: DotfileResource[];
  commands?: CommandResource[];
  secrets?: SecretResource[];
}

// The entry point config
interface DachaConfig {
  repoPath: string;           // Absolute path to the dotfiles repo
  target: Profile;            // The fully composed profile for this machine
  params?: ParamDefinition[]; // Parameters to prompt for on first run
  sync?: {
    enabled: boolean;
    debounceMs?: number;      // Default 2000
  };
  update?: {
    enabled: boolean;
    intervalHours?: number;   // Default 24
  };
}

// Parameter definition вАФ declared in config, prompted interactively
interface ParamDefinition {
  name: string;               // Key used to reference the value (e.g. "gitEmail")
  message: string;            // Prompt message shown to user
  type: "text" | "confirm" | "select";
  default?: string | boolean;
  choices?: string[];         // For "select" type
}

// Resolved params available to config at evaluation time
type Params = Record<string, string | boolean>;

// Built-in path variables, resolved per platform
interface Paths {
  home: string;       // e.g. /Users/ebrouwer.dev or /home/user
  configDir: string;  // ~/.config (XDG_CONFIG_HOME)
  dataDir: string;    // ~/.local/share (XDG_DATA_HOME)
  cacheDir: string;   // ~/.cache (XDG_CACHE_HOME)
  tmpDir: string;     // /tmp or $TMPDIR
  repoDir: string;    // The dotfiles repo root
}

// Output of synth вАФ the resolved desired state
interface ResolvedState {
  platform: Platform;
  resources: ResolvedResource[];  // In topological order
  metadata: {
    generatedAt: string;
    repoPath: string;
    profileChain: string[];   // e.g. ["base", "desktop", "macos", "my-macbook"]
  };
}

interface ResolvedResource {
  id: string;
  type: Resource["type"];
  action: Record<string, unknown>;  // Type-specific action details
  dependsOn: string[];
  contributedBy: string;
}
```

### Profile Resolution (profile.ts)

Profile merging follows a depth-first, left-to-right traversal of the `extends` chain. For each resource type (packages, dotfiles, commands, secrets), arrays are concatenated from parent to child. When a child declares a resource with the same `id` as a parent, the child's version replaces the parent's.

```
Algorithm: resolveProfile(profile)
  1. If profile has no extends, return profile as-is
  2. For each parent in profile.extends (left to right):
     a. Recursively resolve the parent
  3. Merge all resolved parents into a single base (left to right, later wins)
  4. Overlay the current profile on top (child wins on id conflicts)
  5. Tag each resource with contributedBy = profile.name
  6. Return merged profile
```

### Dependency Graph (graph.ts)

The graph is built from two sources:
1. Explicit `dependsOn` references between resources
2. Implicit dependencies from `{{output.resourceId.key}}` template references in dotfile content and command strings

```
Algorithm: buildGraph(resources)
  1. Create adjacency list from all resources
  2. For each resource, scan for {{output.X.Y}} patterns in string fields
     - Add edge: this resource depends on resource X
  3. Add edges from explicit dependsOn arrays
  4. Run cycle detection (DFS with coloring)
     - If cycle found, throw with the cycle path
  5. Topological sort (Kahn's algorithm)
  6. Return ordered resource list
```

### Resource Executors (resources/*.ts)

Each resource type implements a common interface:

```typescript
interface ResourceExecutor<T extends Resource> {
  // Check if the resource is already in the desired state
  check(resource: T, platform: Platform): Promise<boolean>;
  // Apply the resource (install, copy, run, decrypt)
  apply(resource: T, platform: Platform, outputs: OutputStore): Promise<ResourceResult>;
}

interface ResourceResult {
  status: "applied" | "skipped" | "failed";
  outputs?: Record<string, string>;
  error?: string;
}

// OutputStore holds all captured outputs, keyed by resource id
type OutputStore = Map<string, Record<string, string>>;
```

**PackageExecutor**: Checks if package is installed (e.g. `brew list <pkg>`, `dpkg -l <pkg>`). If not, runs the install command. Captures the installed version as an output.

**DotfileExecutor**: Compares source file hash with destination file hash. If different, prompts user (unless `--yes` flag). Copies file, creating parent dirs. If `template: true`, interpolates `{{output.X.Y}}` references from the OutputStore before writing.

**CommandExecutor**: Runs the `check` command first. If it exits 0, skips. Otherwise runs `run`. If `captureOutput` is set, captures stdout into the OutputStore.

**SecretExecutor**: Checks if destination exists with correct hash. If not, runs `age -d -i <identity> <source>` and writes to destination with specified permissions.

### Sync Daemon (sync/daemon.ts)

```
Process: SyncDaemon
  1. Load dacha.config.ts to get list of managed dotfiles
  2. Start Deno.watchFs on all destination paths
  3. On file change event:
     a. Reset debounce timer for that file (2s default)
     b. When timer fires:
        i.   Copy destination вЖТ repo source path
        ii.  git add <source path>
        iii. git commit -m "auto-sync: update <filename>"
        iv.  git push
        v.   If push fails, log error, mark as pending
  4. Periodically (every 60s), check for pending pushes:
     a. Check network connectivity (DNS resolve or HTTP HEAD)
     b. If online, retry git push
  5. Log all activity to platform log location
```

The daemon is managed as:
- macOS: `~/Library/LaunchAgents/dev.dacha.sync.plist` (generated by `dacha sync start`)
- Linux: `~/.config/systemd/user/dacha-sync.service` (generated by `dacha sync start`)

### Update Checker (update/checker.ts)

```
Process: UpdateChecker (runs on schedule)
  1. git fetch origin
  2. Compare local HEAD with origin/main
  3. If behind:
     a. Get list of changed files (git diff --name-only HEAD..origin/main)
     b. Send notification with file list
     c. Write state to ~/.local/share/dacha/update-pending.json
  4. If up to date, exit silently
```

Scheduled via:
- macOS: `~/Library/LaunchAgents/dev.dacha.update.plist` with `StartInterval`
- Linux: `~/.config/systemd/user/dacha-update.timer`

### Parameter Prompting (params.ts)

The config declares parameters via `params` in `DachaConfig`. On first run (or when the lock file is missing), the system prompts interactively using Deno's stdin + a prompt library (e.g. Cliffy's prompt module).

```
Flow:
  1. Read ~/.config/dacha/params.lock.json
  2. For each ParamDefinition in config:
     a. If value exists in lock file вЖТ use it
     b. If missing вЖТ prompt user interactively
  3. Write updated lock file (atomic write)
  4. Return Params record to synthesizer
```

The lock file format:

```json
{
  "version": 1,
  "createdAt": "2026-03-19T12:00:00Z",
  "params": {
    "gitEmail": "user@example.com",
    "hostname": "my-macbook",
    "enableGaming": true
  }
}
```

`dacha params reset [name]` deletes the named param (or all) from the lock file, causing re-prompting on next run.

The config entry point receives params as an argument:

```typescript
// dacha.config.ts
export default function config(ctx: { platform: Platform; params: Params; paths: Paths }): DachaConfig {
  return {
    repoPath: "~/.dotfiles",
    params: [
      { name: "gitEmail", message: "Git email address?", type: "text" },
      { name: "hostname", message: "Machine hostname?", type: "text", default: ctx.platform.os },
      { name: "enableGaming", message: "Install gaming apps?", type: "confirm", default: false },
    ],
    target: buildProfile(ctx),
    // ...
  };
}
```

### Install Script (install.sh)

A POSIX shell script that:

```
1. Detect OS (uname -s) and arch (uname -m)
2. Map to binary name: dacha-{darwin,linux}-{arm64,x64}
3. Download from GitHub releases: https://github.com/<org>/dacha/releases/latest/download/<binary>
4. Verify checksum (download sha256sums.txt, compare)
5. Install to ~/.local/bin/dacha, chmod +x
6. Print PATH advice if ~/.local/bin not in PATH
7. If --repo <url> argument provided, run: dacha init <url>
```

Usage:
```sh
# Install only
curl -fsSL https://raw.githubusercontent.com/<org>/dacha/main/install.sh | sh

# Install + bootstrap
curl -fsSL https://raw.githubusercontent.com/<org>/dacha/main/install.sh | sh -s -- --repo https://github.com/user/dotfiles
```

### Init Command (init.ts)

```
dacha init <github-url> [--path ~/.dotfiles]
  1. If path doesn't exist: git clone <url> <path>
  2. If path exists and is a git repo: git pull
  3. Write ~/.config/dacha/config.json with { repoPath: <path> }
  4. Run dacha apply
  5. If sync enabled in config: run dacha sync start
  6. If update enabled in config: install update checker schedule
```

The global config at `~/.config/dacha/config.json` stores the repo path so all other commands (`apply`, `sync`, `update`, `status`) can find the dotfiles repo without the user specifying it each time.
